# Prompt Optimizer for Laser Cleaning Article Generator

Optimize the provided **Laser Cleaning Article Generator Prompt** to be clear, modular, reusable, and executable by Grok, preserving all instructions in intent and exact form. The optimizer is streamlined for architectural simplicity, tailored to the article generator’s JSON schemas, template engines, Chart.js configurations, and dynamic data sourcing. All content is processed within this prompt, with the target prompt pasted in the **Target Prompt** section.

---

## Workflow Overview

1. **Input**: Define and validate input parameters.
2. **Analysis**: Identify issues in the target prompt (e.g., redundancy, ambiguity, Grok compatibility).
3. **Enhancement**: Improve clarity, modularity, and reusability while preserving all instructions.
4. **Validation**: Verify compliance with requirements, prioritizing instruction preservation.
5. **Output**: Generate the optimized prompt in a single, complete Markdown block.

---

## Shared Rules

These rules apply across all workflow steps to ensure consistency and Grok compatibility:

1. **Instruction Preservation**: Retain all original instructions, including JSON schemas, template functions, Chart.js configurations (e.g., colors, labels, axes, legends, titles, fonts, sizes, grid lines), and validation rules, unless explicitly consolidated for redundancy.
2. **Formatting Retention**: Preserve exact formatting (e.g., `<table class="table data-table">`, `<div class="content-section content-section-[theme]">`, Chart.js syntax) as specified in the target prompt.
3. **Grok Integration**:
   - Store validated inputs (e.g., `[Target_Prompt]`, `[Subject]`, `[Article_Type]`) in Grok’s memory for cross-session reuse, referencing `[Session_ID]`.
   - Use real-time search for context-specific data (query: “recent prompt optimization techniques for laser cleaning article generation” or “industry standards for `[Subject]` laser cleaning”).
   - Leverage Grok’s canvas panel for Chart.js rendering validation.
4. **Error Handling**: Log errors and proceed unless critical (e.g., missing `[Target_Prompt]`, invalid JSON schema), in which case halt with an error message.
5. **Output Control**: Print the optimized prompt only after all processing (analysis, enhancement, validation) is complete, in a single Markdown block.

## Analysis

1. **Instruction Inventory**:
   - Parse `[Target_Prompt]` into categories: Parameters, JSON Schema, Template Functions, Validation Rules, Data Sourcing, Content Rules, Formatting.
   - Generate an **Instruction Inventory Table** (Category, Instruction, Purpose, Location).

     | JSON Schema | `"GlobalConfig.SectionTemplates"` | Defines section structure | GlobalConfig |

2. **Issue Detection**:
   - **Ambiguity**: Flag vague terms (e.g., “sufficient depth”) and clarify using real-time search (query: “industry standards for laser cleaning `[Subject]`”) or defaults (e.g., 2–3 paragraphs for “depth”).
   - **Redundancy**: Identify repeated rules (e.g., formatting preservation across modules) for consolidation into [Shared Rules](#shared-rules).
   - **Contradictions**: Flag conflicting rules (e.g., “halt” vs. “log” for errors) and resolve by prioritizing stricter action.
   - **Grok Compatibility**: Identify infeasible validations (e.g., visual rendering checks) and replace with syntax checks (e.g., regex for `<div class="content-section content-section-[theme]">`).
   - **Article Generator Specificity**: Flag missing instructions for JSON validation, Chart.js integration, or dynamic data sourcing (e.g., ASTM/ASNT standards).

3. **Documentation**:
   - Note assumptions (e.g., “Assumes `[Target_Prompt]` includes JSON schema”).
   - Log issues for enhancement (e.g., “Redundant formatting rules in GlobalConfig and ValidationRules”).

## Enhancement

1. **Clarity**:
   - Simplify language (e.g., “ensure data variety” instead of “metric diversity”).
   - Use numbered sections with descriptive headings (e.g., “JSON Schema Configuration”).
   - Standardize terms (e.g., “section” vs. “module”).

2. **Modularity**:
   - Group related instructions into sections: Parameters, JSON Schema, Template Engine, Validation, Data Sourcing, Output.
   - Reference [Shared Rules](#shared-rules) for formatting, error handling, and Grok integration.
   - Example: Replace repeated “preserve `<table class="table data-table">`” with “[Shared_Rules].Formatting”.

3. **Redundancy Reduction**:
   - Consolidate repeated instructions (e.g., formatting, Chart.js settings) into [Shared Rules](#shared-rules).
   - Use placeholders (e.g., `[Section_Type]`) for reusable templates (e.g., validation checklists).

4. **Article Generator Specificity**:
   - **JSON Schema**: Validate and simplify schema structure, ensuring required fields (e.g., `SectionTemplates`, `ValidationRules`) are explicit.
   - **Template Engine**: Preserve template functions (e.g., `renderParagraph`, `renderChart`) and ensure Chart.js syntax compatibility.
   - **Data Sourcing**: Explicitly define queries (e.g., “recent laser cleaning settings for `[Subject]`”) and fallback standards (e.g., ASTM D7091).
   - **Chart.js Integration**: Retain all chart settings (e.g., `chartColors`, `chartFontSize`, `data.labels`) and validate canvas rendering via Grok’s canvas panel.

5. **Reusability**:
   - Use generic variables (e.g., `[Subject]`, `[Article_Type]`) with defaults in [Input](#input).
   - Ensure templates (e.g., validation checklists) are reusable across article types.

6. **Grok Compatibility**:
   - Ensure Grok can parse Markdown, JSON, HTML, and JavaScript (e.g., Chart.js scripts).
   - Use real-time search for missing data (e.g., “laser cleaning innovations `[Subject]` 2024–2025”).
   - Validate Chart.js rendering via Grok’s canvas panel, ensuring unique `canvasId`.

7. **Prioritization**:
   - Prioritize instruction preservation over brevity, ensuring all JSON fields, template functions, and validation rules are retained.
   - Apply brevity only for redundant rules, consolidating into [Shared Rules](#shared-rules).

     | Original Instruction | Revised Form | Location |

2. **Formatting Compliance**:
   - Validate HTML/CSS (e.g., `<div class="content-section content-section-[theme]" data-theme="[theme]">`) using regex.
   - Verify Chart.js syntax (e.g., `canvasId`, `chartColors`) via Grok’s canvas panel.

3. **Grok Feasibility**:
   - Ensure validations are executable (e.g., regex for HTML, JSON schema checks).
   - Replace infeasible rules (e.g., visual rendering) with syntax checks.

4. **Completeness**:
   - Confirm all sections (Parameters, JSON Schema, Template Engine, Validation, Output) are processed before printing.

5. **Checklist**:
   - **Critical (Halt)**:
     - All instructions preserved, including JSON schemas, template functions, Chart.js settings.
     - Formatting retained (e.g., `<table class="table data-table">`, Chart.js configurations).
     - Variables defined with defaults.
     - No hard-coded user inputs.
     - Prompt printed only when complete.
   - **Warning (Log and Proceed)**:
     - Validations feasible for Grok.
     - Modules reusable across article types.
