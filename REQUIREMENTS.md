The Z-Beam generator project has the following requirements:

Overall:   
Generate content in the form of mdx files that can be used in the Z-Beam Next.js website, next to this /genernator directory and called z-beam-test-push.

Content Categories:
contentCategory: ["application","author","material","region","thesaurus"] // The current mdx files should have contentCategory:"material" under 'title'.

Application Content Category: 
These are freshly generated mdx files with article content that will have internal crosslinks to the thesaurus tags if they occur in the article.

Author Content Category: 
An author ID is manually assigned to mdx file content, appearing only in the metadata.

Material Content Category: 
The material is manually assigned in run.py and guides the content generation.  It includes internal crosslinks to the thesaurus tags if they occur in the article.

Region Content Category: 
The region is manually assigned in run.py and guides the content generation.  It includes internal crosslinks to the thesaurus tags if they occur in the article.

Thesaurus Content Category: 
A list of terms relevant to laser cleaning.  These are used as Tags in the mdx files.   A given mdx file can have multiple tags.  When the mdx file is generated these are formatted as links.

How content is generated:
A new mdx page is generated by this project, composed of 'sections' that are individually generated and assembled at the end.  

How sections are generated:
There is prompting in json that describes the section requirements as a prompt.  This is sent to a detection provider that retutns content which is then further processed.

How content is processed:
The detection provider has devised iterative and dynamic feedback services in this genernator to help the content meet requirements.  This optimization is iterated and scored.

Training:
An interactive training mode allows the user to evaluate the quality of generated content and provide feedback to the production generator.  The generator will use this feedback to improve the quality of the generated content.  Optimization values are updated.

Production:
resulting mdx files are sent to the ../z-beam-test-push/app/content directory.

QUALITY RULES:

## Core System Principles

### 1. NO FALLBACKS POLICY
- **CRITICAL RULE**: The system must NEVER generate content when production content already exists
- **TRAINING CONSTRAINT**: Training modes must ONLY use existing, pre-generated production content  
- **FAILURE MODE**: Fail fast with clear error messages if content doesn't exist
- **USER WORKFLOW**: If content doesn't exist, user must run production generation first
- **RATIONALE**: Ensures training evaluates actual production output, not synthetic content

### 2. Anti-Hardcoding Compliance  
- **MANDATORY**: All configuration values MUST come from `GlobalConfigManager`
- **FORBIDDEN**: Magic numbers, hardcoded URLs, API keys, or model names in code
- **REQUIRED**: Use `get_config()` methods for ALL parameter access
- **VALIDATION**: Anti-hardcoding tests must pass before deployment

### 3. Production Content Integrity
- **CONTENT SOURCE**: Training systems evaluate final, optimized production content only
- **PIPELINE COMPLETION**: Content must have completed full iteration cycles and optimization
- **AUTHENTIC METRICS**: Word budgets and detection scores must reflect production pipeline results
- **SECTION EXTRACTION**: Must work only with existing content structure - no approximation

### 4. Clean Architecture
- **SEPARATION**: Strict separation of domain, application, and infrastructure layers
- **DEPENDENCY INJECTION**: For all services and repositories  
- **INTERFACE-BASED**: Design for testability and modularity
- **CONTAINER MANAGEMENT**: Use DI container for service resolution

### 5. Dynamic Optimization Requirements
- **DYNAMIC CONFIG MANAGEMENT**: All optimization parameters MUST be managed through `GlobalConfigManager`
- **NO STATIC THRESHOLDS**: AI detection, natural voice, and temperature values must be dynamically updateable
- **TRAINING-DRIVEN UPDATES**: System must automatically adjust optimization parameters based on user feedback
- **INTELLIGENT DEFAULTS**: Provide optimal default values when user hasn't specified custom settings
- **RUNTIME VALIDATION**: Ensure optimization values stay within safe operational ranges
- **CENTRALIZED ACCESS**: All optimization values accessed via `get_config()` methods - no direct value access
- **FEEDBACK INTEGRATION**: Training feedback must automatically update production optimization parameters
- **PARAMETER PERSISTENCE**: Optimization changes must persist across system restarts
- **AUDIT TRAIL**: All optimization parameter changes must be logged with reasoning
- **MANUAL OVERRIDE**: Users can still set specific optimization values when needed

### 6. Configuration Hierarchy (Enforced Order)
1. **User Explicit Values**: Values explicitly set in `USER_CONFIG` (highest priority)
2. **Dynamic Training Updates**: Values adjusted based on training feedback and performance
3. **Intelligent Defaults**: Optimal values provided by `GlobalConfigManager`
4. **System Fallbacks**: Hardcoded defaults only as absolute last resort

### 7. Optimization Parameter Categories (All Must Be Dynamic)
- **Detection Thresholds**: AI detection, natural voice detection limits
- **Temperature Settings**: Content, detection, improvement, summary, metadata generation
- **Quality Bounds**: Word budgets, section requirements, validation criteria
- **Iteration Controls**: Retry limits, improvement cycles, optimization attempts
- **Performance Tuning**: API timeouts, rate limits, batch sizes

## Violation Consequences
- **Development**: Immediate fix required before code review
- **Runtime**: Fast failure with clear recovery instructions
- **No silent degradation**: Never substitute content without explicit user request in the system
- **Hardcoded Parameters**: Automatic detection tools must identify and flag violations
- **Config Bypass**: Direct parameter access without `GlobalConfigManager` is forbidden

## Required Dynamic Optimization Commands

### Configuration Inspection
- `python3 show_config.py` - Display current optimization settings with clear explanations
- `python3 workflow.py show-recommendations` - Show pending optimization recommendations

### Training Integration  
- `python3 train.py` - Interactive training that collects user feedback on production content
- `python3 workflow.py apply-training` - Apply training insights to optimization parameters automatically
- Training must update optimization parameters based on user feedback patterns

### Hardcoding Detection
- `python3 workflow.py detect` - Identify hardcoded configuration values in codebase
- `python3 workflow.py autofix` - Automatically fix common configuration violations
- Anti-hardcoding tests must pass before any deployment

### Production Workflow Requirements
- Training modes must ONLY evaluate existing production content from `/output` directory
- Content generation must use dynamically optimized parameters from `GlobalConfigManager`
- All optimization parameter access must go through `get_config()` methods
- System must fail fast if attempting to train on non-existent production content

## Quality Assurance Requirements

### Automated Testing
- **Anti-hardcoding tests**: Must detect and prevent hardcoded configuration values
- **Configuration validation**: Ensure optimization parameters stay within safe ranges  
- **Training integration tests**: Verify feedback properly updates optimization parameters
- **Production content tests**: Confirm training uses actual production output only

### Documentation Standards
- All optimization parameters must be documented with purpose and acceptable ranges
- Training workflows must be documented with clear no-fallbacks policy
- Configuration hierarchy must be documented and enforced

## Dynamic Optimization Validation Tests

### Required Test Categories
1. **Parameter Access Validation**
   - Verify all optimization parameters accessed through `GlobalConfigManager.get_config()`
   - Detect any direct parameter access or hardcoded values
   - Ensure fallback behavior only occurs for true system emergencies

2. **Training Feedback Integration**
   - Confirm user feedback updates optimization parameters correctly
   - Validate parameter changes persist across system restarts
   - Test parameter bounds enforcement (no invalid values)

3. **Production Content Integrity**
   - Verify training modes load actual production content from `/output` directory
   - Ensure no synthetic content generation during training evaluation
   - Confirm section extraction works only with existing content structure

4. **Configuration Hierarchy Enforcement**
   - Test that user explicit values override dynamic updates
   - Verify dynamic training updates override intelligent defaults
   - Ensure intelligent defaults override system fallbacks
   - Validate manual override capability without breaking automation

### Test Implementation Requirements
- All tests must be runnable via `python3 test_runner.py dynamic-optimization`
- Tests must fail fast with clear error messages when violations detected
- Integration tests must verify end-to-end training-to-production feedback loop
- Performance tests must validate optimization parameter effectiveness

### Success Criteria
- **Zero Hardcoding**: No hardcoded optimization parameters in production code
- **Dynamic Updates**: Training feedback automatically improves optimization parameters
- **Production Fidelity**: Training evaluates identical content to what users receive
- **Configuration Transparency**: All optimization settings viewable and explainable
